## How to Work With the Squad

**New to the team?** Start here:

1. **Read this file** (you're here! ✓)
2. **Read `../team.md`** — Get the roster and official roles
3. **Skim `../decisions.md`** — Understand what's been decided and why
4. **Pick an agent** based on your expertise:
   - Frontend work → **Rusty's domain**
   - Backend/integration work → **Linus's domain**
   - Quality/testing work → **Livingston's domain**
   - Architecture decisions → **Danny's domain**
   - Work prioritization → **Ralph's domain**
   - Documentation → **Scribe's domain**

5. **Read that agent's personality.md** — Understand their working style

## Contributing Tips by Agent

### If You're Working with Danny
- Think systemically. What's the impact on the overall platform?
- Be ready to defend decisions with logic, not intuition.
- Get buy-in early; once Danny decides, it's decided.

### If You're Working with Rusty
- Ship early iterations for feedback.
- Care about UX as much as you care about functionality.
- Accessibility and dark mode aren't optional.

### If You're Working with Linus
- Write specs together before building.
- Don't rush integration work; it compounds debt.
- Ask about error handling and edge cases early.

### If You're Working with Livingston
- Test-driven development is non-negotiable.
- Edge cases matter. A lot.
- Quality gates aren't obstacles; they're your safety net.

### If You're Working with Scribe
- Explain your reasoning, not just your decision.
- Help her document patterns so they're reusable.
- Decisions should be written down, always.

### If You're Working with Ralph
- Tell him when you're blocked immediately.
- Celebrate shipped work—Ralph feeds on momentum.
- Help him prioritize ruthlessly.
